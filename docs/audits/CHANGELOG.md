- **Manual backup export profile binding (VERIFY-102):** Electron backup encryption now requires a one-time, five-minute export lease bound to the sender's main-process profile session. The renderer and main process both reject profile changes, payloads carry encrypted profile provenance, non-default exports omit legacy unscoped/default records, and main-process validation rejects token reuse, mismatched metadata, or cross-profile records before encryption.
- **Encrypted Conversation Vault profile isolation (VERIFY-101):** The historical default vault layout and encryption AAD remain unchanged, while non-default profiles now use validated profile roots for records, manifests, journals, attachments, and memory indexes. Vault/index caches and write queues are profile-keyed, non-default AAD is bound to the profile, and list/get/save/delete/archive/search/context/rebuild/open-folder IPC paths derive storage authority from the sender's WebContents session.
- **Legacy desktop-chat profile isolation (VERIFY-100):** Historical default-profile JSON remains in `chat-history/`, while non-default profiles now persist under validated profile subdirectories. Legacy chat list/get/save/delete and remote-sync apply derive the profile from the sender's WebContents session, identical logical IDs no longer collide across profiles, and non-default sessions cannot inspect or trigger default-only legacy migration.
- **Main-process credential/password administration isolation (VERIFY-099):** Venice, Jina, and fallback-provider key status/set/delete/test paths plus profile-password set/clear now derive their secure-store profile from the sender's authenticated WebContents session. Electron profile deletion requires activating the target first, the Profile panel exposes destructive administration only for the active profile, and deletion sweeps credentials for every registered fallback provider.
- **Main-process provider-use profile isolation (VERIFY-098):** Ordinary and streaming Venice IPC requests now replace renderer-supplied profile selectors with the sender's WebContents session before validation. Jina request key lookup uses the same session authority, so forged or invalid renderer profile IDs cannot select another profile's credential or force a request denial.
- **Main-process profile-session task isolation (VERIFY-097):** Desktop profile switches now activate a WebContents-scoped main-process session for both protected and unprotected profiles. Persistent background-task creation, snapshots, lists, mutations, removals, and broadcasts derive their profile from that session, ignore forged create-profile IDs, and return a generic not-found result for cross-profile mutation attempts.

- **BUG-006 (payload):** Every failed Venice request produced two entries in the diagnostics log: one with `error: ""` from the initial HTTP dispatch, and a second with the actual message from the retry/catch path. The redundant dispatch has been removed; the initial entry now carries the fully resolved error.
- **BUG-007 (payload):** Legacy persisted settings could still carry boolean `webSearch` values (`true`/`false`), which mapped to invalid API payloads and recurring `400` schema errors. Settings ingestion now coerces legacy values to `on`/`off`/`auto`, and chat payload construction enforces the same normalization.
- **BUG-008 (payload):** Desktop `/augment/text-parser` uploads were unstable because `veniceFetchDesktop` serialized `FormData` but still sent the raw body over IPC. The request now correctly sends the serialized multipart payload, restoring reliable file-upload request construction.

### Security
- Explicitly disabled plaintext API-key fallback on macOS.
- Documented web proxy forbidden-header stripping and proxy-root rejection.
- Added Venice.ai TOS, privacy, and API documentation coverage for public releases.
